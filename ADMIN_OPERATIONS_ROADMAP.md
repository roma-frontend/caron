## Правила Ведения

- Перед началом нового пункта отметить его как `in progress`.
- После реализации отметить как `done` и кратко написать, какие файлы изменены.
- Если пункт требует решения владельца бизнеса, оставить статус `blocked` и вопрос.
- Не смешивать несколько крупных пунктов в одну задачу, чтобы изменения было легко проверять.

## Легенда

- `[ ]` not started
- `[~]` in progress
- `[x]` done

## Этап 2. Отдельные Финансовые Суммы

Статус: [x]

Задача: разделить деньги на понятные бизнес-показатели.

Показатели:

- Оплачено
- Ожидает оплаты
- Отменено
- Возвраты
- Чистый доход
- Средний чек
- Наличные
- Банковский перевод
- Idram
- EasyPay
- Карта

Правила расчёта:

- Доход считать только по `paymentStatus === 'paid'` и `status !== 'cancelled'`.
- Отменённые считать отдельно по `status === 'cancelled'`.
- Ожидает оплаты считать по `paymentStatus === 'awaiting'` и `status !== 'cancelled'`.
- Возвраты считать по `paymentStatus === 'refunded'` или отдельному будущему полю возврата.

Ожидаемый результат:

- Видно, сколько реально денег пришло.
- Видно, сколько денег зависло в ожидании оплаты.
- Видно, сколько потеряно из-за отмен.

Возможные файлы:

- `src/app/admin/orders/page.tsx`
- `src/app/admin/page.tsx`

Реализовано:

- Добавлена финансовая разбивка на странице заказов по выбранному периоду.
- Добавлены суммы: оплачено, ожидает оплаты, отменено, возвраты, чистый доход, средний чек.
- Добавлены суммы по способам оплаты: наличные, банковский перевод, Idram, EasyPay, карта.
- Файлы: `src/app/admin/orders/page.tsx`

## Этап 3. История Действий Заказа

Статус: [~]

Задача: сохранять историю изменений каждого заказа.

Что логировать:

- Изменение статуса заказа
- Изменение статуса оплаты
- Отмена заказа
- Возврат товара на склад
- Повторное открытие отменённого заказа
- Комментарий менеджера
- Кто сделал действие
- Когда сделал действие

Пример записи:

```text
2026-06-16 14:32 | Admin | status: pending -> cancelled | Товар возвращён на склад
```

Ожидаемый результат:

- Понятно, кто и когда отменил заказ.
- Понятно, вернулся ли товар на склад.
- Легче разбирать спорные ситуации с клиентами.

Возможные изменения:

- Новая таблица Convex: `orderEvents`
- Новая query для истории заказа
- UI-блок истории в карточке/деталях заказа

Возможные файлы:

- `convex/schema.ts`
- `convex/orders.ts`
- `src/app/admin/orders/page.tsx`

Реализовано:

- Добавлена таблица `orderEvents` в `convex/schema.ts`.
- Запись событий при создании заказа (`created`) и при изменении статуса/оплаты (`status_changed`, `cancelled`, `reopened`, `payment_changed`).
- Query `getOrderEvents` для получения истории по заказу.
- Блок истории в `OrderCard`: кнопка с иконкой часов, раскрывающийся список событий с датой, именем админа и описанием.
- Файлы: `convex/schema.ts`, `convex/orders.ts`, `src/app/admin/orders/page.tsx`

## Этап 8. Проблемные Заказы

Статус: [ ]

Задача: автоматически показывать заказы, которые требуют внимания.

Проблемные случаи:

- Доставлен, но не оплачен
- Оплачен, но отменён
- Заказ долго в статусе `pending`
- Заказ долго ожидает оплату
- Нет телефона или email
- Сумма подозрительно маленькая или 0
- Товар в заказе больше не существует

Ожидаемый результат:

- Менеджер видит список проблем без ручного поиска.
- Меньше забытых заказов и ошибок в оплате.
